System

Platform info, tech stack, architecture, security, GDPR, and roadmap.

summarize Executive Summary expand_more

1. Scope & Objective

SignFlow is a self-hosted, white-label newsletter signup platform operated by WingVibes. It enables the creation and management of fully branded, embeddable signup forms across multiple properties — without reliance on third-party subscription services, recurring SaaS fees, or ceding control of subscriber data to an external vendor. The business that runs this portal owns everything: the audience, the data, and the relationship.

The core objective is full data sovereignty. WingVibes controls what is collected, how it is stored, who can access it, and when it is deleted. No external platform can lock in subscribers, impose limits on list size, or change pricing terms — because the platform itself is the asset.

2. Functionality & Flexibility

Administrators can create and manage multiple signup forms — each independently branded to any property or campaign, with its own field set, confirmation flow, welcome email, and embed configuration. Forms support a wide variety of input types beyond a simple email box: date pickers, age selectors, icon and image tile selectors, sliders, and rich content blocks including video, preference toggles, and spin-wheel incentives. The visual form builder includes a live preview, and completed designs can be saved as reusable templates to accelerate new launches.

Every form can be embedded on any external website via a single script tag, or accessed as a standalone page. Subscriber lists are exportable at any time as CSV or JSON — the data is always retrievable in full, with no export fees or gating. The platform is built to grow alongside the business: new field types, integrations, and audience segments can be added without structural changes to the system.

3. Security

Security is built in at every layer, not retrofitted. Admin access requires authentication through Auth0 (an industry-standard identity provider) with role-based access control — only credentialed administrators can reach the admin panel. All state-changing admin actions are protected against cross-site request forgery (CSRF), all user-submitted content is sanitised before storage or display, and rate limiting is applied per form and per IP to prevent abuse of the public signup endpoints.

Subscriber data is stored in a private, encrypted PostgreSQL database on AWS — not in shared cloud services or accessible flat files. Media uploads go directly to a private Amazon S3 bucket. No subscriber data is shared with advertising networks, analytics platforms, or any third party beyond the transactional email provider configured by WingVibes.

4. Enterprise-Grade Architecture

SignFlow is built on the same technology stack used by large-scale production platforms: Node.js on AWS Lightsail, PostgreSQL 16 (AWS-managed cloud database), Amazon S3 for media storage, Nginx as a reverse proxy with TLS, and Let's Encrypt for automatic SSL certificate renewal. Sessions are persisted in the database and survive server restarts; the process manager (PM2) automatically restarts the application after any failure or reboot — the platform is designed to stay up without manual intervention.

The admin interface is a single-page application with no build pipeline. Updates are deployed in seconds. The architecture is intentionally compact to operate efficiently at the current scale, while remaining fully legible and straightforward to hand to an external developer or agency if requirements expand.

5. Key Risks & Mitigations

The primary operational risk is the single-server model: if the AWS Lightsail instance becomes unavailable, the platform goes offline. This is mitigated by automated process restarts via PM2, continuous database backups via Lightsail's managed snapshot system, and the low recovery complexity — the application can be restored on a fresh instance from the repository and environment variables alone. The second risk is key-person dependency: all architecture decisions, configuration, and deployment knowledge are documented in this system panel and the project README specifically to make the platform operable without its original developer.

On the data side, the primary risk is a breach of the subscriber database. Mitigations include encrypted storage at rest (Lightsail managed DB default), private network access only (no public database endpoint), strong session secrets, strict input validation on all subscriber-facing routes, and Content Security Policy headers that block injection of unauthorised scripts. GDPR compliance obligations — right to erasure, consent audit logging, data export, and a three-category cookie consent manager with script blocking — are fully implemented in the current codebase.

6. Roadmap

Near-term priorities are completing the security hardening phase (adding CAPTCHA to the public signup route and automated dependency vulnerability scanning). GA4 and Meta Pixel integration with a GDPR-compliant consent manager are now live — analytics (GA4) and marketing (Meta Pixel) scripts are consent-gated and injected dynamically only after the visitor opts in. Medium-term, the platform will gain webhook and API integrations — enabling subscriber data to flow automatically into email service providers (Mailchimp, Klaviyo) or CRM platforms (HubSpot) without manual exports, reducing operational overhead as audience volume grows.

Longer term, the roadmap includes a full REST API with API-key authentication, multi-user admin access with role-based permissions, double opt-in email confirmation flows, and configurable data retention policies. All of these extend the current architecture rather than replace it — the foundation is already in place.

7. GDPR & Data Protection

SignFlow is built with data protection as a first-class requirement. Every subscriber signup captures a consent timestamp and IP address as a permanent audit record — evidence that consent was freely given at a specific moment in time. Each subscriber receives a unique, private token granting them direct self-service access to update their preferences, unsubscribe from individual newsletters, or request complete deletion of their data — with no need to contact WingVibes directly. All deletion requests result in a hard, permanent removal from the database, satisfying the right to erasure under UK and EU GDPR.

A GDPR-compliant cookie consent manager is displayed on every direct-load form page. Visitors must explicitly opt in before any analytics (GA4) or marketing (Meta Pixel) scripts are loaded — no tracking scripts fire until consent is granted. Visitors can withdraw or change their preferences at any time via the "Cookie settings" link. Consent is versioned and timestamped in localStorage.

WingVibes, as the business operating this platform, is the data controller. Subscriber data is never shared with third parties without the visitor's explicit cookie consent. Analytics and marketing data is only transmitted to Google / Meta if the visitor opts in. Data is held exclusively on private AWS infrastructure within the EU (Paris region). The platform's export capability allows a full data package to be produced for any subject access request within minutes. Remaining compliance obligations — maintaining a Record of Processing Activities and executing a Data Processing Agreement with the configured SMTP provider — sit with WingVibes as operator and are outside the platform's technical scope.

8. AI-Assisted Development & Developer Handover

SignFlow is developed using Claude (Anthropic's AI coding assistant) as the primary development tool. Every feature and architectural decision is implemented through a structured, documented process: requirements are stated in plain language, a technical plan is reviewed and approved before any code is written, and changes are verified against the live codebase before deployment. The full development history — including the intent behind each decision and the rationale for every design choice — is captured in session transcripts, not buried in commit messages. Any developer brought in can understand not just what was built, but why, without a handover meeting.

The codebase is deliberately maintained in two files — server.js and admin/index.html — with no build pipeline, no framework abstractions, and no compiled output. Any Node.js developer can open either file and immediately orient themselves. This system panel documents the full architecture, security model, environment configuration, and deployment process. The combination of AI-assisted development practices and self-contained documentation means WingVibes can onboard a new developer, engage an agency, or transfer ownership of the platform without any loss of institutional knowledge.

XSS prevention — Plan C: admin subscriber table

Previously, subscriber emails and custom field values were injected directly into tr.innerHTML, and action buttons used onclick="fn('${s.email}')" — a pattern vulnerable to JS injection via crafted email addresses.

Fix 1 — data escaping: All subscriber values (email, status, date, custom fields, IDs) are now wrapped in escHtml() before being placed in innerHTML.

Fix 2 — event delegation: All onclick attribute handlers removed. Buttons carry data-action, data-id, and data-email attributes. A single IIFE-wrapped document.addEventListener('click', …) listens at document level, checks that the target is inside #sub-tbody, reads the data-* attributes, and dispatches to the appropriate function — eliminating all string interpolation in event handler context.

Before: onclick="adminUnsubscribe('${s.id}','${s.email}')"
After: data-action="unsub" data-id="…" data-email="…" + delegated listener

X-Frame-Options — Plan F

Helmet sets X-Frame-Options: SAMEORIGIN globally, complementing frame-ancestors 'self' in the CSP for older browsers that do not support CSP Level 2.

The embed route (GET /:slug/embed) overrides framing headers. If embedAllowedDomains is empty (default) the form can be embedded anywhere:

X-Frame-Options: ALLOWALL
Content-Security-Policy: … frame-ancestors *; …

If embedAllowedDomains is configured (e.g. ["mysite.com","app.co"]), the embed route checks the request Origin / Referer header and returns 403 if it doesn't match. The CSP is tightened accordingly:

Content-Security-Policy: … frame-ancestors https://mysite.com https://app.co; …

All other CSP directives (script-src, style-src, frame-src, etc.) are preserved on the embed route — only frame-ancestors changes.

Input validation — Plan G: Joi schemas

All user-submitted data on the public subscribe route is validated by a dynamically-built joi schema before any DB access occurs.

analytics Embed & Analytics Architecture expand_more

SignFlow handles analytics in two fundamentally different ways depending on how the form is accessed. Understanding this distinction is important for correct setup and GDPR compliance.

Direct-load page (/your-slug)

SignFlow owns the page, so it controls the full lifecycle — including consent and script loading.

  • A GDPR-compliant consent manager is shown on first visit. No third-party scripts fire until the visitor explicitly opts in.
  • GA4 and Meta Pixel IDs are stored in Settings → Analytics & Tracking. They are embedded as a <script type="application/json" id="sf-tc"> data block — not executed at load time.
  • The consent manager JS reads that block and dynamically injects gtag.js / fbevents.js only after the visitor grants the relevant category.
  • On successful submission, the page fires gtag('event','generate_lead') and fbq('track','Lead') — both conditional on window.gtag / window.fbq existing (i.e. consent was given).

Embedded form (/your-slug/embed via iframe)

SignFlow does not own the page — it is a guest inside the host site's iframe. The host site controls consent.

  • The embed page contains no consent manager and no tracking scripts whatsoever.
  • The GA4 / Pixel IDs configured in Settings are not used in the embed context.
  • On successful submission, the iframe fires a postMessage to the parent: { type: 'sf-success', slug: '...' }.
  • embed.js (running on the host page, outside the iframe) listens for that message and calls window.gtag / window.fbq if they exist.
  • Those globals only exist if the host site's own consent manager has already loaded them. If the visitor rejected cookies on the host site, neither global exists and nothing fires.

Overview

A planned feature: a floating AI assistant panel in the admin that answers questions with awareness of the current page, active form config, and live data. Architecture combines RAG (semantic search over help docs via pgvector) with Tool Use / Function Calling so Claude can both explain how things work and inspect the user's actual live state before answering.

Why RAG + Tool Use together?

TechniqueAnswersLimitation alone
RAG"How does X work?"Can't see live data — gives generic advice
Tool use"What is happening?"No knowledge of docs — can't explain features
Combined"Why isn't X working for me?"—

Example: "Why is my embed blank?" → search_docs("embed blank page") retrieves the relevant doc chunks → get_form("wing-signup") fetches actual embedAllowedDomains value → Claude answers with the user's specific settings, not generic steps.

Key: search_docs is itself a Claude tool — Claude decides when to search and with what query, and can retry with different terms if the first result is weak.

RAG pipeline — pgvector (zero new infrastructure)

-- One migration
CREATE EXTENSION IF NOT EXISTS vector;
CREATE TABLE doc_embeddings (
  id        SERIAL PRIMARY KEY,
  chunk_id  TEXT UNIQUE,       -- e.g. "help-embed-1", "system-security-2"
  content   TEXT,              -- 200–400 token plain-text chunk
  embedding vector(1536),      -- from text-embedding-3-small or Voyage AI
  metadata  JSONB              -- { section, tab, tags[] }
);
CREATE INDEX ON doc_embeddings USING ivfflat (embedding vector_cosine_ops);
// scripts/embed-docs.js — run once at deploy; re-run when docs change
// 1. Parse Help + System tab HTML into plain-text chunks (~300 tokens each)
// 2. Embed each chunk:
//    • OpenAI text-embedding-3-small  ($0.02 / 1M tokens — pennies for ~150 chunks)
//    • OR Voyage AI voyage-3-lite     (Anthropic's recommended embedding partner)
// 3. Upsert into doc_embeddings (ON CONFLICT chunk_id DO UPDATE)

async function search_docs(query, limit = 4) {
  const qEmbed = await embedText(query);
  const { rows } = await pool.query(
    `SELECT content, metadata, 1 - (embedding <=> $1) AS similarity
     FROM doc_embeddings ORDER BY embedding <=> $1 LIMIT $2`,
    [pgvector.toSql(qEmbed), limit]
  );
  return rows;
}

Server route — agentic loop with SSE streaming

app.post('/api/admin/ai-chat', adminAuth, async (req, res) => {
  const { message, context, history = [] } = req.body;
  // context = { currentTab, currentFormSlug, activeModal, panelTab }
  // history = prior [ {role,content} ] turns for multi-turn awareness

  res.setHeader('Content-Type', 'text/event-stream');
  res.setHeader('Cache-Control', 'no-cache');

  const system =
    `You are a concise SignFlow admin assistant. Max 3–4 sentences unless genuinely ` +
    `needed. Reference exact setting names and their location in the UI. ` +
    `Context: tab="${context.currentTab}", form="${context.currentFormSlug||'none'}", ` +
    `modal="${context.activeModal||'none'}". ` +
    `Always call tools before answering — never guess from memory alone.`;

  const messages = [
    ...history,
    { role: 'user', content: message }
  ];

  // Agentic loop — Claude may call tools multiple times before final answer
  while (true) {
    const response = await anthropic.messages.create({
      model: 'claude-sonnet-4-5', max_tokens: 1024,
      system, tools, messages
    });

    if (response.stop_reason === 'end_turn') {
      for (const block of response.content)
        if (block.type === 'text')
          res.write(`data: ${JSON.stringify({ text: block.text })}\n\n`);
      break;
    }

    if (response.stop_reason === 'tool_use') {
      messages.push({ role: 'assistant', content: response.content });
      const results = [];
      for (const block of response.content) {
        if (block.type !== 'tool_use') continue;
        const result = await executeTool(block.name, block.input);
        results.push({ type: 'tool_result', tool_use_id: block.id,
                       content: JSON.stringify(result) });
      }
      messages.push({ role: 'user', content: results });
      // loop → Claude sees results, may call more tools or produce final answer
    }
  }

  res.write('data: [DONE]\n\n');
  res.end();
});

Embedding options

        
Every form has two embed methods. Open the Embed tab to see the generated code.

        

          
MethodHow it works

          
iFrameA standard <iframe> pointing to /slug/embed. Specify a fixed height or enable auto-resize. No JavaScript required on the host page.

          
JS WidgetA single <script> tag that loads /slug/embed.js. The script injects a self-resizing iframe into a target <div data-signflow="slug"> element. The iframe auto-resizes as content changes.

        

        
Embed page styling

        
The embed page (/slug/embed) renders without the outer page chrome (no background, no cookie banner). It uses the same design settings as the main form but with slightly smaller typography and tighter padding for compact integration.

        
Auto-resize

        
The JS widget communicates iframe height via postMessage, so the host page never shows a scrollbar inside the embed — the iframe grows to fit its content.

Site settings

        
    
          
  • Form title — shown in browser tab and email subject line
    • 
          
  • Slug — URL path segment (e.g. my-newsletter → /my-newsletter). Must be unique and URL-safe. Reserved slugs: admin, auth, api, embed, preferences.
    • 
          
  • Privacy policy URL — used in the GDPR disclaimer and cookie consent banner links
    • 
          
  • Cookie banner text — the explanatory text shown in the consent banner. Supports rich text (bold, links). Leave blank to use the built-in default notice.
    • 
        

        
Cookie Consent Manager

        
SignFlow includes a GDPR-compliant cookie consent manager on every direct-load form page. Key behaviours:

        
    
          
  • Three categories — Necessary (always active), Analytics (GA4 — opt-in), Marketing (Meta Pixel — opt-in). Category rows only appear when the corresponding ID is configured in Analytics & Tracking.
    • 
          
  • Script blocking — GA4 and Meta Pixel scripts are never injected at page load. They are loaded dynamically by client-side JS only after the visitor explicitly grants consent for that category.
    • 
          
  • Three-button UI — Accept all, Necessary only (equal prominence, no dark patterns), and Manage preferences (expands granular toggles — all off by default).
    • 
          
  • Persistent choice — consent is stored as JSON in localStorage under key sf_consent with a version number and timestamp. Previously consented visitors have scripts applied immediately on load without seeing the banner again.
    • 
          
  • Withdraw anytime — a Cookie settings link appears at the bottom-right of the page after first consent, allowing visitors to reopen the manager and change or withdraw their choices.
    • 
          
  • Embeds excluded — the consent banner does not appear inside embedded iframes. Analytics events for embeds are handled via the host page's existing GA4/Pixel via postMessage.
    • 
        

        
hCaptcha

        
Enable bot protection per-form. Enter your hCaptcha site key and secret key (from hcaptcha.com). When enabled, a CAPTCHA challenge appears above the submit button.

        
Analytics & Tracking (GA4 + Meta Pixel)

        
Two tracking methods are supported:

        
    
          
  • Direct-load forms (wingvibes.com/your-slug) — enter a GA4 Measurement ID (e.g. G-XXXXXXXXXX) and/or a Meta Pixel ID in Settings → Analytics & Tracking. The scripts are injected into the form page and fire generate_lead / Lead events on successful submission.
    • 
          
  • Embedded forms (iframe via embed.js) — no IDs needed. On submission the embed script fires a generate_lead / Lead event directly into the host page's existing GA4 and Meta Pixel instances (if present), using postMessage. It also dispatches a signflow:success DOM event on the parent window for custom integrations: window.addEventListener('signflow:success', e => console.log(e.detail.slug)).
    • 
        

        
Branding assignment

        
Link a branding to a form here. The live form will use the branding's visual design while keeping form-specific settings (slug, fields, sections) intact.

        
Submission deduplication

        
By default SignFlow prevents the same email from subscribing twice. The Submission deduplication card lets you change this behaviour:

        
    
          
  • Allow duplicate emails — when toggled on, the same email can submit multiple times as long as the chosen key fields differ
    • 
          
  • Unique key fields — tick which fields together form the uniqueness check (e.g. Email + First name). A submission is blocked only when every ticked field matches an existing active subscriber exactly.
    • 
        

        
Example: with Email + First name as the key, alice@example.com / Alice and alice@example.com / Bob are treated as two distinct subscribers.
